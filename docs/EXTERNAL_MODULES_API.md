# API Внешних Модулей (External Modules API)

Внешние модули — это сторонние скрипты или приложения (например, Telegram-боты, n8n, Home Assistant), которые могут взаимодействовать с Ядром (Dom.ru Integration Hub) для получения событий с домофона и отправки команд (например, открытия двери).

## Авторизация (Токен)
Для работы с API вашему модулю нужен **Токен**. 
Токен можно сгенерировать в веб-интерфейсе Ядра (раздел "Настройки" -> "Внешние модули"). 

Авторизация передается в каждом HTTP-запросе одним из способов:
1. Через заголовок: Authorization: Bearer <ваш_токен> (рекомендуется)
2. Через query-параметр: ?token=<ваш_токен>

---

## 1. Получение событий (Webhook / WebSocket)

Модуль может получать события от Ядра (например, когда кто-то звонит в домофон) двумя способами: через Webhooks или через WebSockets. Модуль сам решает, какой способ ему удобнее.

### Способ А: Настройка Webhook (HTTP POST)
Если вы хотите, чтобы Ядро отправляло POST-запросы на ваш сервер при наступлении событий, вам нужно зарегистрировать свой Webhook URL в Ядре.

**Запрос (от Модуля к Ядру):**
POST /api/modules/me/webhook

**Заголовки:**
Authorization: Bearer <ваш_токен>
Content-Type: application/json

**Тело:**
`json
{
  "webhookUrl": "https://ваш-домен.com/api/webhook"
}
`

**Формат события (от Ядра к Модулю):**
Ядро будет отправлять POST-запросы на ваш webhookUrl с таким телом:
`json
{
  "event": "incoming_call",
  "payload": {
    "callId": "...",
    "intercomId": "...",
    "timestamp": 1690000000000
  }
}
`

### Способ Б: Подключение по WebSocket
Если вашему скрипту проще держать постоянное соединение (например, локальные Python-скрипты), вы можете подключиться по протоколу WebSocket.

**URL для подключения:**
ws://<адрес_ядра>/api/modules/ws?token=<ваш_токен>

После успешного подключения Ядро начнет транслировать все события домофона в виде JSON-сообщений.

---

## 2. Действия и Команды

Модуль может отдавать команды Ядру, например, открыть дверь домофона.

### Открыть дверь (Action Open)
**Запрос:**
POST /api/modules/actions/open

В заголовке передаем Authorization: Bearer <ваш_токен>.
В случае успеха вернется JSON { "success": true }.

## 3. Schema-Driven UI (Настройки Модуля)

Внешние модули могут динамически запрашивать у пользователя настройки через UI Ядра.
Для этого модуль должен зарегистрировать свою Схему Настроек. Ядро нарисует по этой схеме форму, и пользователь введет туда данные.

### Регистрация Схемы
**Запрос:**
POST /api/modules/me/schema

В заголовке: Authorization: Bearer <ваш_токен>

**Тело:**
`json
{
  "instruction": "Для работы бота получите токен у @BotFather в Telegram и вставьте его ниже.",
  "fields": [
    {
      "key": "bot_token",
      "type": "password",
      "label": "Токен бота",
      "required": true,
      "description": "Пример: 123456:ABC-DEF1234ghIkl-zyx57W2v1u123ew11"
    }
  ]
}
`

### Получение сохраненных настроек
Чтобы получить значения, которые пользователь ввел в интерфейсе Ядра, выполните запрос:

**Запрос:**
GET /api/modules/me/settings

В заголовке: Authorization: Bearer <ваш_токен>

В случае успеха Ядро вернет JSON-объект в формате ключ-значение:
`json
{
  "bot_token": "введенный_пользователем_токен"
}
`

## 4. Статусы и Мониторинг (UI State)

Ядро не "пингует" Webhook-модули и не пытается угадать их статус. Модуль **должен сам сообщать Ядру о своем состоянии**. Это состояние (и любые ошибки) мгновенно отображается пользователю в веб-интерфейсе.

### Отправка статуса
**Запрос:**
POST /api/modules/me/status

В заголовке: Authorization: Bearer <ваш_токен>
Content-Type: application/json

**Тело:**
`json
{
  "status": "online", 
  "message": "Модуль готов к работе! Токен Telegram проверен."
}
`

**Доступные статусы (status):**
- online — модуль работает исправно. В UI отображается зеленая галочка "В сети".
- offline — модуль выключен или не настроен. Серая вилка "Не в сети".
- warning — модуль требует внимания пользователя (например, нужно ввести настройки в интерфейсе). Желтый значок "Требуется настройка".
- error — произошла критическая ошибка в логике модуля (например, API Telegram недоступно). Красный значок "Ошибка".

*Параметр message выводится во всплывающей подсказке (tooltip) при наведении мышки на статус модуля в интерфейсе Ядра.*

## 5. Динамическая регистрация Возможностей (Capabilities)

Внешние модули могут регистрировать свои уникальные **Возможности** в Ядре. Каждая зарегистрированная возможность появляется в настройках каждого жильца/пропуска в веб-интерфейсе Ядра в виде переключателя (toggle). Пользователь может включить или выключить данную возможность для конкретного человека (например, "Распознавание лиц" для жильца).

### Регистрация Возможности
**Запрос:**
POST /api/modules/me/register-capability

В заголовке: Authorization: Bearer <ваш_токен>
Content-Type: application/json

**Тело:**
```json
{
  "name": "FACE_RECOGNITION",
  "label": "Распознавание лиц (Face ID)",
  "description": "Автоматическое открытие двери на основе видеоанализа лица посетителя"
}
```

В случае успеха возможность регистрируется, и её статус немедленно синхронизируется с базой данных людей. Пользователи могут индивидуально активировать или деактивировать "FACE_RECOGNITION" для жильцов, гостей и курьеров.

## 6. Мониторинг статуса конкретных сущностей (Entity-Level Status)

Когда модуль обрабатывает сущности (например, профили жильцов для Face ID), он может сообщать статус обработки для каждого конкретного человека. Это состояние (например, "Загрузка фото", "Ошибка кодирования", "Успешно") отображается красивыми бейджами в веб-интерфейсе прямо в карточке человека.

### Обновление статуса сущности
**Запрос:**
POST /api/modules/me/entity-status

В заголовке: Authorization: Bearer <ваш_токен>
Content-Type: application/json

**Тело:**
```json
{
  "entityType": "person",
  "entityId": "user-123456",
  "status": "processing",
  "message": "Декодирование файла (45672 байт)..."
}
```

**Доступные статусы для сущностей:**
- `processing` (Желтый бейдж) — профиль обрабатывается модулем (например, идет скачивание или расчет эмбеддингов).
- `success` (Зеленый бейдж) — профиль успешно синхронизирован и активен в модуле.
- `error` (Красный бейдж) — произошла ошибка при обработке (например, на фотографии не найдено лицо). Текст ошибки выводится пользователю.
- `disabled` (Серый бейдж) — профиль временно отключен.

## 7. Изолированное хранилище Модулей (Isolated Persistent Storage)

Ядро предоставляет внешним модулям изолированное дисковое хранилище. Каждый модуль может сохранять, читать и удалять бинарные данные (например, фотографии, обработанные изображения, файлы логов или кастомные кэши) по уникальным ключам. Данные сохраняются на сервере в `/data/modules-storage/<module_id>.json` в сериализованном формате.

### Чтение данных по ключу
**Запрос:**
GET /api/modules/storage/:moduleId/:key

*Доступен без токена для отрисовки ресурсов (например, через стандартные теги `<img>` в UI).*

### Запись данных
**Запрос:**
POST /api/modules/storage/:moduleId/:key

В заголовке: Authorization: Bearer <ваш_токен>
Content-Type: application/json (или multipart/form-data)

**Тело:**
Любые бинарные данные (например, JPEG изображение лица жильца). 
*Примечание: При записи фотографии жильца Ядро автоматически переводит статус обработки этого жильца в `success` (или `processing` до успешной обработки).*

### Удаление данных
**Запрос:**
DELETE /api/modules/storage/:moduleId/:key

В заголовке: Authorization: Bearer <ваш_токен>

## 8. WebSocket-интеграция в реальном времени (Пространство имен `/modules`)

Для высокоскоростных локальных интеграций (таких как модули видеоаналитики Face ID) рекомендуется постоянное соединение по протоколу Socket.IO.

### Подключение к пространству имен `/modules`
Модули должны подключаться к выделенному пространству имен `/modules`.

**Авторизационный хэндшейк (auth handshake):**
```json
{
  "token": "ваш-уникальный-токен-модуля",
  "moduleId": "уникальный-id-модуля-в-интерфейсе"
}
```

### Разрешение конфликтов подключений (Conflict Resolution)
Ядро реализует строгий контроль сессий:
1. К одному токену модуля в каждый момент времени может быть подключен **только один активный сокет**.
2. При подключении нового экземпляра с тем же токеном (например, после перезапуска контейнера модуля), Ядро **автоматически завершает предыдущую устаревшую сессию** сокета, отправляя событие `"error"` с кодом `"duplicate_connection"`. Это гарантирует отсутствие "двойных" обработчиков.

### События Ядра (Сервер -> Модуль)
- `people_updated` — генерируется при любых изменениях в базе данных людей Ядра (добавление жильца, удаление, изменение расписания или переключение возможностей). Получив это событие, модуль должен выполнить синхронизацию людей.
- `module_data_updated` — сигнализирует об обновлении локального хранилища модуля.

### События Модуля (Модуль -> Сервер)
- `update_status` — передача текущего статуса модуля (замена HTTP POST `/api/modules/me/status`).
- `update_entity_status` — передача статуса обработки человека (замена HTTP POST `/api/modules/me/entity-status`).
- `action_open_door` — отправка запроса на открытие двери.
