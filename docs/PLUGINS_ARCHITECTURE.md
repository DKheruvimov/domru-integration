# Архитектура Плагинов (Smart Intercom Hub)

Система Дом.ru Intercom интеграции расширяется за счет модульной архитектуры. Это позволяет сообществу (и пользователю) подключать кастомные функции без изменения ядра приложения. 

Проект поддерживает (или планирует поддерживать) два типа расширений:
1. **External Modules (Внешние модули)** — Уже реализовано.
2. **Internal Plugins (Внутренние плагины)** — Запланировано к реализации.

---

## 1. External Modules (Реализовано)

Внешние модули — это самостоятельные приложения (например, скрипты на Python для распознавания лиц `dom-ru-eye`, интеграции на Go, Home Assistant и т.д.), которые крутятся на другом железе или в других контейнерах.

**Как это работает:**
- Пользователь генерирует токен (API ключ) в веб-интерфейсе во вкладке «Внешние модули».
- Токены хранятся в `data/modules.json` (управляются через `server/modules-manager.ts`).
- **События (WebSocket):** Внешний скрипт подключается к нашему серверу по адресу `wss://ВАШ_ДОМЕН/modules?token=ВАШ_ТОКЕН`.
  - При звонке ядро отправляет в этот сокет событие `incoming_call`.
  - Полезная нагрузка (Payload) содержит `deviceId`, `placeId` и актуальный `snapshotUrl`, по которому можно скачать фото звонящего.
  - Состояние подключения (Online/Offline) транслируется в React-интерфейс в реальном времени.
- **Действия (REST API):** Если скрипт принял решение (например, лицо совпало), он отправляет POST запрос на `/api/modules/actions/open` с заголовком `Authorization: Bearer ВАШ_ТОКЕН` и `deviceId` в теле запроса. Ядро само находит SIP-сессию и открывает дверь.

---

## 2. Internal Plugins (В планах для будущих сессий)

Внутренние плагины — это легковесные JavaScript/TypeScript скрипты, которые будут выполняться непосредственно внутри Node.js процесса Ядра. Это идеальное решение для легких интеграций (например, Telegram бот, уведомления в ВК, проброс статуса в MQTT).

**План реализации (Что нужно сделать ИИ агенту в будущем):**

1. **Менеджер плагинов (`server/plugin-manager.ts`)**
   - Должен сканировать директорию `/plugins/` на наличие папок со скриптами `index.js/ts`.
   - При старте Ядра, менеджер загружает эти файлы и вызывает их дефолтную функцию инициализации.
   - Динамическая перезагрузка (Hot Reloading) плагинов без рестарта всего Ядра (опционально).

2. **Core API (Контракт)**
   - В функцию инициализации плагина передается объект `PluginAPI`.
   - Подписки: `api.on('incoming_call', callback)`, `api.on('door_opened', callback)`.
   - Вызовы: `api.actions.openDoor(deviceId)`, `api.actions.getSnapshot(deviceId)`.
   - Хранилище: `api.getConfig()` — плагины не должны использовать свои `.env` файлы. Все настройки плагина хранятся централизованно в `data/plugins.json` и управляются Ядром.

3. **Интеграция в UI (Строгий дизайн)**
   - **ВАЖНОЕ ПРАВИЛО:** Визуал настроек плагинов должен на 100% совпадать со стилистикой сервиса. Цветовая мешанина запрещена.
   - Плагины должны экспортировать схему своих настроек (JSON Schema или интерфейс), а React-приложение (Ядро) будет автоматически отрисовывать инпуты, переключатели и кнопки в стиле `zinc` и `emerald/red` (Tailwind).
   - Вкладка «Плагины» в настройках, где пользователь сможет включать/выключать их и вводить ключи (например, `bot_token` для Telegram).
   - **Capability Flags (Влияние плагинов на основной UI):** Плагины не могут напрямую встраивать свой HTML/JS в основной интерфейс (например, во вкладку «Люди»). Вместо этого они регистрируют свои «Возможности» при старте Ядра: `api.registerCapability('FACE_RECOGNITION')`. Основной UI (React) содержит заранее заготовленные, но скрытые компоненты. Если Ядро передает на фронтенд флаг `FACE_RECOGNITION`, React-приложение автоматически прячет расписание для Жильцов и показывает элементы загрузки фото для авто-открытия по лицу.

4. **Паттерн Gateway (Шлюз для внешних сервисов)**
   - Плагины **никогда** не общаются напрямую с внешними платформами (например, Яндексом) и не хранят токены пользователей.
   - Ядро выступает единым шлюзом (Gateway).
   - **Пример (Умный дом Яндекса):**
     - Плагин регистрирует виртуальный датчик в Ядре: `api.yandex.registerVirtualSensor({ id: 'face_sensor', type: 'button' })`.
     - Ядро автоматически объединяет этот датчик с реальными домофонами и отдает Яндексу единым списком при Device Discovery.
     - При срабатывании триггера (например, распознано лицо), плагин вызывает `api.yandex.triggerSensor('face_sensor', 'click')`. Ядро берет на себя подпись запроса OAuth-токеном и отправку State Report в Яндекс.
   - Это гарантирует безопасность, изолирует плагины от изменений сторонних API и обеспечивает лучший UX (пользователю достаточно привязать всего один аккаунт/навык).

## 3. Отложенные идеи (Roadmap)

### 3.1. Кастомные голосовые оповещения Алисы (Proactive TTS)
**Идея:** Динамическое произнесение Алисой индивидуальных фраз при входящем звонке (например, «Звонит курьер», «Пришёл Иван Иванов»).
**Статус:** Отложено.
**Причина:** 
- Официальный путь (Сценарии Умного дома + виртуальные датчики) требует ручной настройки статических сценариев пользователем и не поддерживает передачу динамического текста с сервера.
- Неофициальный путь (Yandex Station Local API / неофициальные библиотеки) позволяет передавать динамический текст, но является нестабильным, требует логина по паролю/кукам в сервисы Яндекса и подвержен поломкам при изменениях API Яндекса. Учитывая наличие физической трубки домофона, разработка неофициального и требующего поддержки решения признана нецелесообразной.
К этому вопросу можно вернуться в будущем, если Яндекс откроет официальное API для отправки произвольного TTS на колонки.

### 3.2. Раздельное хранилище данных для плагинов
**Идея:** Изолировать данные, которыми управляют плагины (например, параметры распознавания, фотографии жильцов), от основного системного хранилища и истории событий.
**План реализации:** 
- У каждого плагина должно быть собственное изолированное хранилище (namespace), например в `data/plugins/{plugin_name}/`.
- Ядро будет предоставлять плагину методы `api.storage.get()` и `api.storage.set()`, чтобы плагины не модифицировали общие файлы вроде `people.json` напрямую. Это защитит ядро от поломок при сбоях в логике плагинов.

### 3.3. Комбинирование способов автооткрытия (Паттерн 2FA)
**Идея:** Позволить пользователям включать несколько критериев автооткрытия одновременно, создавая сложные сценарии (например, «Расписание» + «Распознавание по лицу»).
**План реализации:**
- В интерфейсе карточки человека (жильца/гостя) будут добавлены независимые переключатели для каждой зарегистрированной «Capability» плагинов.
- Плагин будет определять свои критерии (кому доступна фича).
- Ядро будет использовать логику «И» (AND) при проверке условий: дверь откроется по лицу только в том случае, если текущее время также попадает в разрешенные интервалы расписания. Это работает как своеобразная двухфакторная аутентификация для физического доступа.
