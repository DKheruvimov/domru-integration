# AI-Driven Development Workflow

Этот документ содержит правила и workflow для любых AI-агентов, работающих с данным проектом.

## Основной процесс (Workflow)

1. **Контекст в первую очередь**: Перед началом любой задачи прочитайте `ARCHITECTURE.md` и `KNOWN_ISSUES.md`, чтобы не изобретать велосипед и не ломать существующую логику (особенно SIP и видео-стриминг).
2. **Анализ кода**: Если нужно изменить существующую функцию, воспользуйтесь поиском по файлам. Всегда проверяйте интерфейсы перед их модификацией.
3. **Общение с пользователем**: Если требования непонятны, задавайте уточняющие вопросы до написания кода. Не делайте предположений о сложной бизнес-логике.
4. **Шаги выполнения**:
   - Планирование (создание `implementation_plan.md` если задача масштабная).
   - Выполнение (создание `task.md` и поэтапное закрытие чекбоксов).
   - Валидация (проверка компиляции/типов).
   - Создание отчета (`walkthrough.md`).

## Правила написания кода

### Стилизация и UI
- **Vanilla CSS**: Основной проект использует Vanilla CSS. Не вводить Tailwind CSS в React-компоненты.
- **Tailwind CDN**: Используется **исключительно** в standalone HTML-странице `server/views/oauth-consent.html`. Не распространять на остальной проект.
- **Анимации**: Используется библиотека **motion** (Framer Motion): `AnimatePresence`, `motion.div` и т.д. для модальных окон и переходов.
- **Модальные окна (No Alert/Confirm)**: Ввиду работы приложения в iFrame-песочнице, **запрещено** использовать `window.confirm()` или `window.alert()`. Создавайте кастомные анимированные React-диалоги.
- **Дизайн**: Современный (glassmorphism, плавные анимации). Тёмная и светлая тема поддерживаются.

### TypeScript
- Все новые файлы — `.ts` или `.tsx` со строгой типизацией. Минимизировать `any`.
- **Общие типы**: `Person`, `ScheduleRule`, `SharedCredentials` — в `shared/types.ts`. Не дублировать определения.

### Анонимизация
- **Строго запрещено** указывать личные домены. Все примеры, документация и комментарии — `yourdomain.com`.

### Отсутствие заглушек (No Placeholders)
- Не оставлять комментарии вида `// insert logic here` или `// TODO`. Пишите рабочий код.

## Real-time обновления (WebSocket)

В проекте реализована система WebSocket через **Socket.IO** для обновления UI в реальном времени.

### Категорически запрещено
- Использовать `setInterval`-polling для обновления состояния UI.
- Создавать внешние webhook-эндпоинты для обновления фронтенда.

### Доступные broadcast-функции (`server/ws-manager.ts`)

| Функция | Событие | Когда использовать |
|---------|---------|-------------------|
| `broadcastAutoOpenStatusChanged()` | `auto_open_status_changed` | Изменился статус авто-открытия: включение/выключение тумблера, изменение расписания, истечение временной задачи, удаление карточки гостя |
| `broadcastSipLogAdded(log)` | `sip_log_added` | Новая запись в SIP-логах (для `SipLogsViewer`) |
| `broadcastIncomingCall(login, details?)` | `sip_incoming_call` | Входящий SIP-звонок в домофон |
| `broadcastDoorOpened(deviceId, source, details)` | `door_opened` | Дверь открыта любым способом |

### Клиентская подписка
Фронтенд подключается через синглтон из `src/socket.ts`:
```typescript
import { getSocket } from "../socket";
const socket = getSocket();
socket.on("auto_open_status_changed", () => { /* обновить данные */ });
```

## Работа с SIP-привязками

### Постоянные привязки (`permanentBindings`)
При старте сервера `sip-init.ts` автоматически регистрирует SIP-учётные данные для всех сохранённых аккаунтов. Эти привязки **не снимаются** после звонка — они нужны для перехвата входящих вызовов и создания снапшотов.

### Временные задачи (`activeTasks`)
Создаются при включении авто-открытия (UI тумблер, Алиса). Имеют `expiresAt` и опционально `maxOpens`/`opensRemaining`.

### Синхронизация карточек и SIP-задач
При изменении карточки пользователя (удаление, отключение тумблера `enabled`):
1. Проверить, является ли пользователь временным (`id.startsWith("temp-")`).
2. Если да — вызвать `disableAutoOpenByDevice(deviceId)` для сброса SIP-сессий.
3. Вызвать `broadcastAutoOpenStatusChanged()` для обновления UI.

## Московское время (MSK)

При написании серверных таймеров, триггеров и проверок расписаний:
- Всегда использовать `getMskTime()` из `server/people-manager.ts`.
- **Запрещено** использовать `new Date().getHours()` / `.getDay()` без MSK-нормализации.
- Все поля `expiresAt` хранятся в абсолютных UTC миллисекундах (`Date.now()`).
- Формат расписаний: `startTime`/`endTime` — строки `"HH:MM"` в **московском** времени.

## Форматы времени (Unix Timestamp)

API Дом.ru возвращает `timestamp` как **строки** с Unix-секундами (например, `"1782749097"`).
- Парсить в число: `Number(timestamp)`.
- Конвертировать в мс: `Number(timestamp) * 1000`.
- Сравнивать напрямую: `Math.abs(A - B) <= maxDiffMs`. Без модульной арифметики.

## Яндекс Алиса и роли

- Два типа временных пользователей: `guest` и `courier`.
- Всегда явно прокидывать роли при `activateAutoOpen`.
- Пользователи запускают команды через **Сценарии Умного дома** («Выполнить команду»).
- При активации через Алису — создавать и SIP-задачу, и визуальную карточку в People.

## Структура данных на диске (`data/`)

| Файл | Формат | Назначение |
|------|--------|-----------|
| `tokens.json` | `Record<UUID, SharedCredentials>` | OAuth-токены Яндекс ↔ Дом.ru |
| `people.json` | `Person[]` | Жильцы, гости, курьеры + расписания |
| `openings.json` | `DoorOpeningRecord[]` | Журнал открытий (30 дней) |
| `settings.json` | `AppSettings` | Задержки авто-открытия |
| `sip_tasks.json` | `AutoOpenTask[]` (serialized) | Активные SIP авто-открытия |
| `snapshots.json` | `SipSnapshotEntry[]` | Индекс снапшотов |
| `snapshots/` | `*.jpg` | JPEG-файлы снапшотов камер |

## Развертывание и проксирование

- Проект часто разворачивается за Reverse Proxy (Cloudflare, TurboFlare). Express настроен с `app.set("trust proxy", true)`.
- Входящий IP — IP прокси. Исходящий трафик к API Дом.ru — реальный IP сервера.
- Если провайдер (Cloud.ru) банит исходящие коннекты, нужен Forward Proxy/VPN на уровне ОС.

## Работа с терминалом

- **Инструменты**: Запрещено использовать `cat`, `grep`, `sed` внутри bash-команд, если доступны встроенные инструменты IDE (`read_file`, `grep_search`, `replace_file_content`).
- **Запуск скриптов**: Проверяйте, не запущен ли уже сервер (Vite) в фоновом режиме.
