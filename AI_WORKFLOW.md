# AI-Driven Development Workflow

Этот документ содержит правила и workflow для любых AI-агентов, работающих с данным проектом.

## Основной процесс (Workflow)

1. **Контекст в первую очередь**: Перед началом любой задачи прочитайте `ARCHITECTURE.md` и `KNOWN_ISSUES.md`, чтобы не изобретать велосипед и не ломать существующую логику (особенно SIP и видео-стриминг).
2. **Анализ кода**: Если нужно изменить существующую функцию, воспользуйтесь поиском по файлам. Всегда проверяйте интерфейсы перед их модификацией.
3. **Общение с пользователем**: Если требования непонятны, задавайте уточняющие вопросы до написания кода. Не делайте предположений о сложной бизнес-логике.
4. **Шаги выполнения**:
   - Планирование (создание `implementation_plan.md` если задача масштабная).
   - Выполнение (создание `task.md` и поэтапное закрытие чекбоксов).
   - Валидация (проверка компиляции/типов).
   - Создание отчета (`walkthrough.md`).

## Правила написания кода

- **Анонимизация**: В проекте **строго запрещено** указывать личные домены. Все примеры, документация и комментарии должны использовать обезличенные домены (например, `yourdomain.com`).
- **Стилизация**: Проект использует Vanilla CSS. Использовать TailwindCSS можно только если он явно запрошен пользователем или уже присутствует в целевом компоненте. Дизайн должен быть современным (glassmorphism, плавные анимации).
- **TypeScript**: Все новые файлы должны иметь `.ts` или `.tsx` расширение и строгую типизацию. Не используйте `any` без крайней необходимости.
- **Отсутствие заглушек (Placeholders)**: При написании кода не оставляйте комментарии вида `// insert logic here`. Пишите полностью рабочий код.
- **Интерфейс без нативных окон (No Alert/Confirm)**: Ввиду работы приложения в iFrame-песочнице, **категорически запрещено** использовать `window.confirm()` или `window.alert()`. Всегда создавайте кастомные анимированные модальные диалоги подтверждения на базе Tailwind CSS и Framer Motion (`AnimatePresence`).
- **Московское время (MSK)**: При написании или изменении любых серверных таймеров, триггеров и проверок активности расписаний, всегда преобразуйте время хоста к Московскому часовому поясу (UTC+3) с помощью готового хелпера `getMskTime()`. Не используйте нативные методы `getHours()`, `getDay()` объекта `new Date()` без учета смещения MSK.
- **Синхронизация карточек и SIP-задач**: При изменении статуса карточки пользователя (сохранение, удаление, переключение тумблера `enabled`), всегда явно проверяйте, не является ли пользователь временным гостем/курьером (ID `temp-*`). В случае его удаления или отключения, необходимо немедленно вызвать `disableAutoOpenByDevice(deviceId)` для сброса активных SIP-сессий и обновления индикаторов камер в UI.
- **Форматы времени (Unix Timestamp)**: API Дом.ru может возвращать время (`timestamp`) в виде **строк** с Unix-секундами (например, `"1782749097"`). Всегда проверяйте тип и парсите такие строки в числа, умножая на 1000 для работы с миллисекундами. Обе отметки (`Date.now()` и распарсенный timestamp API) находятся в формате абсолютного UTC-времени, поэтому при их сравнении используйте прямое вычитание (`Math.abs(A - B)`) без модульной арифметики и обходов часовых поясов.

## Работа с терминалом

- **Инструменты**: Запрещено использовать `cat`, `grep`, `sed` внутри bash-команд, если доступны встроенные инструменты IDE (например, `read_file`, `grep_search`, `replace_file_content`).
- **Запуск скриптов**: Всегда проверяйте, не запущен ли уже сервер (Vite) в фоновом режиме, прежде чем пытаться запустить его снова.
