# Архитектура проекта Dom.ru Integration

Данный документ описывает структуру и ключевые компоненты проекта для быстрой адаптации AI-агентов и новых контрибьюторов.

## Стек технологий
- **Frontend**: React, TypeScript, Vite, Vanilla CSS + Tailwind (если включен), Lucide React.
- **Backend (BFF)**: Node.js, Express, TypeScript (используется как Vite middleware в dev-режиме и как standalone server в prod).
- **Video Handling**: FFmpeg (для транскодирования аудио в AAC и мультиплексирования), hls.js (плеер), mpegts.js (fallback).

## Основные директории
- `src/components/`: React компоненты фронтенда.
- `src/domru-js/`: SDK для взаимодействия с внутренним API myhome.proptech.ru.
- `server/`: Бэкенд логика, проксирование видео и SIP интеграция.

## Ключевые компоненты

### 1. SDK (`src/domru-js/`)
Реализует REST-клиент к инфраструктуре Дом.ru.
- Особенности: Авторизация строится на двух хэшах (MD5 и SHA1).
- Модули разбиты логически: `operators.ts`, `places.ts`, `stream.ts`, `events.ts`.
- Состояние сессии держится через `accessToken`, `refreshToken` и `operatorId`.

### 2. Video Proxy (`server/routes/domruRoutes.ts`)
Обеспечивает обход CORS и проксирование видео с камер.
- Парсит M3U8 плейлисты и на лету подменяет пути к `.ts` сегментам и ключам шифрования на локальные прокси-эндпоинты.
- **Transcoding**: Дом.ru камеры часто отдают аудио в неподдерживаемых форматах или вообще без аудио дорожки. Прокси "на лету" пропускает каждый `.ts` сегмент через FFmpeg (`-c:v copy -c:a aac` или `anullsrc`), чтобы плееры в браузере (и Yandex Smart Home) не падали.

### 3. SIP Manager (`server/sip-manager.ts`)
Реализует функцию "Авто-открытие курьеру".
- Получает SIP учетные данные устройства через `POST rest/v1/places/{placeId}/accesscontrols/{deviceId}/sipdevices`.
- Регистрируется на SIP сервере по протоколу UDP.
- При поступлении `INVITE` мгновенно отвечает `200 OK`, шлет REST-команду на открытие двери, делает `BYE` и снимает регистрацию (чтобы не занимать линию постоянно и вернуть звонки на мобильное приложение пользователя).
- **Синхронизация статуса ожидания**: Статус автооткрытия на камерах контролируется эндпоинтом `/api/domru/sip/auto-open/status`. Иконка "активного ожидания" загорается на камере **исключительно** при наличии активного расписания для гостей или курьеров (жильцов квартиры система не ждёт, так как у них есть свои ключи/приложения).
- **Сброс сессии при удалении/отключении**: При выключении тумблера или удалении временной карточки гостя/курьера на вкладке «Люди», бэкенд шлет сигнал `disableAutoOpenByDevice(deviceId)`, что мгновенно прекращает фоновые SIP-регистрации для данного устройства и сбрасывает статус ожидания в интерфейсе.

### 4. Интеграция с Умным Домом Яндекса
- Файл `server/routes/yandexRoutes.ts` предоставляет необходимые эндпоинты (`/v1.0/user/devices`, `/v1.0/user/devices/query`, `/v1.0/user/devices/action`) для работы навыка Яндекса.
- Камеры транслируются Яндексу через сформированные Proxy URL.

### 5. Управление доступом и расписания (`server/people-manager.ts`)
Управляет списком доверенных лиц, регулярными расписаниями и временными пропусками.
- **Московское время (MSK, UTC+3)**: Все проверки активности расписаний (`isScheduleActive`) на бэкенде нормализуются с помощью хелпера `getMskTime()`. Это гарантирует, что если расписание настроено на «12:00», оно сработает ровно в 12:00 по Москве, независимо от того, где физически развернут контейнер (например, в облаке в европейском или американском регионе).
- **Временные гостевые карты**: Создаются с префиксом `temp-` и временем истечения `expiresAt` в UTC миллисекундах. Интерфейс отображает их дни работы как «Сегодня» вместо перечисления дней недели.
- **Автоматическая фоновая очистка**: При каждом обращении к базе пользователей (`getPeople()`), бэкенд на лету проверяет просроченные временные карточки (роли `courier`/`guest`), удаляет их из файла `people.json` и сохраняет изменения.

## Соглашения о написании кода
- Строгая типизация во всем проекте (TypeScript).
- Коммиты и комментарии пишутся в соответствии с задачами, без указания приватных или личных доменов авторов. Все примеры должны указывать на абстрактные домены (например, `yourdomain.com`).
