# Известные проблемы и архитектурные особенности (Known Issues)

Этот файл содержит базу знаний о проблемах, с которыми мы уже сталкивались. Обязательно к прочтению агентами перед внесением изменений в соответствующие модули.

## 1. SIP Интеграция («Авто-открытие» и перехват звонков)

### 1.1 Постоянные SIP-привязки
- При старте сервера `sip-init.ts` автоматически регистрирует SIP-учётные данные **для всех** устройств (кроме камер) всех сохранённых аккаунтов. Эти привязки (`permanentBindings`) **постоянно активны** и поддерживаются периодической перерегистрацией.
- **Зачем**: Постоянные привязки необходимы для двух функций:
  1. Создание **снапшотов камеры** при каждом входящем звонке в домофон.
  2. **Ручной перехват** — пользователь может нажать «Открыть» в UI во время звонка, и сервер ответит `200 OK` на удерживаемый SIP-диалог.

### 1.2 Ограничение на количество регистраций
- Сервер Дом.ru позволяет только **ОДНУ** активную SIP-регистрацию на устройство. Пока наш сервер зарегистрирован, мобильное приложение Дом.ru **не получает** звонки в домофон.
- **Компромисс**: Это сознательный выбор — функции снапшотов и авто-открытия важнее мобильных пуш-уведомлений.

### 1.3 Синхронизация с базой «Люди»
- Отключение или удаление карточки гостя/курьера на вкладке «Люди» должно **немедленно** вызывать `disableAutoOpenByDevice(deviceId)`, чтобы сбросить активные SIP-задачи и обновить индикаторы на камерах через WebSocket.
- Индикатор активного ожидания загорается **только** при наличии активных расписаний гостей/курьеров (расписания жильцов игнорируются — у них есть свои ключи).

## 2. Авторизация в API (MD5 и SHA1 хэширование)

- Авторизация использует два хэша: SHA1 и MD5 с кастомной солью.
- **Не переписывать** логику генерации `hash1` (SHA1) и `hash2` (MD5) — сервер Дом.ru жёстко валидирует формат. Соль захардкожена в `domru-api/http/client.ts`.

## 3. Видеостриминг, FFmpeg и go2rtc

### 3.1 HLS Proxy (FFmpeg)
- Дом.ru отдаёт камеры в HLS (M3U8). Сегменты `.ts` часто проблемные (нет аудио или проприетарные кодеки).
- **Transcoding**: `server/routes/streamRoutes.ts` проксирует сегменты через FFmpeg и при необходимости транскодирует аудио в AAC.
- Определение наличия аудиодорожки происходит через парсер MPEG-TS (`server/mpegTsParser.ts`).
- **Осторожно с `-copyts`**: Если видео не имеет аудио, мы микшируем с `anullsrc`. Флаг `-copyts` в этом случае использовать **нельзя** — образуется PTS gap, HLS сломается.

### 3.2 go2rtc (WebRTC)
- **go2rtc** — внешний daemon для WebRTC-стриминга с низкой задержкой.
- `server/go2rtc-manager.ts` управляет его жизненным циклом: скачивание бинарника, запуск, автоматический перезапуск при падении, WebSocket-проксирование через `/api/go2rtc/ws`.
- Бинарник go2rtc скачивается автоматически при первом запуске и сохраняется в `bin/`.

## 4. Генерация временных (гостевых) кодов

- API Дом.ru (`getTemporalCodes`) может возвращать ошибки для устройств, где услуга не подключена.
- Фронтенд **не должен** полагаться на успешность генерации кода без обработки ошибок.

## 5. Расписания и временные карточки

### 5.1 Локальное время (Timezone)
- Все проверки `isScheduleActive()` нормализуются через `getServerTime()`, которая использует часовой пояс, заданный в настройках сервера (по умолчанию `Europe/Moscow`).
- Расписание на «12:00» сработает в 12:00 по выбранному часовому поясу, независимо от физического расположения сервера или контейнера.

### 5.2 Ежедневный сброс
- Для роли `courier` с `maxOpens`, поле `opensRemaining` сбрасывается до `maxOpens` при наступлении нового дня (Локальная дата сравнивается с `lastOpenedDate`).

### 5.3 Автоочистка
- Просроченные карточки (ID `temp-*`, `expiresAt < Date.now()`) автоматически удаляются из `people.json` при каждом обращении к `getPeople()`.

## 6. Ограничения iFrame и UI-диалоги

- Приложение работает в iFrame-песочнице. Вызовы `window.confirm()` / `window.alert()` блокируются.
- **Решение**: Кастомные React-компоненты с `AnimatePresence` + `motion.div`.

## 7. Обработка меток времени (Timestamps)

- API Дом.ru возвращает `timestamp` как **строку** с Unix-секундами: `"1782749097"`.
- Конвертация: `Number(timestamp) * 1000` → UTC миллисекунды.
- Сравнение: `Math.abs(A - B) <= maxDiffMs`. Модульная арифметика **запрещена**.

## 8. Обновления состояния в реальном времени (WebSockets)

- В проекте используются **WebSockets** через Socket.IO.
- **Polling запрещён**: Если нужно передать обновление на фронтенд, используйте broadcast-функции из `server/ws-manager.ts`.
- Доступные события: `auto_open_status_changed`, `sip_log_added`, `sip_incoming_call`, `door_opened`.
- Подробная таблица — в `AI_WORKFLOW.md`.

## 9. Развертывание за прокси (Cloud.ru, TurboFlare) и WAF

- Рекомендуется прятать сервер за Reverse Proxy (TurboFlare, Cloudflare). Express настроен с `app.set("trust proxy", true)`.
- **WAF и кастомные заголовки**: Многие Web Application Firewalls (например, Turboflare) автоматически блокируют HTTP-запросы, содержащие слово `password` в названиях заголовков (например, `x-domru-password`), возвращая 403 Forbidden.
- **Решение**: Передача кредов между фронтендом и бэкендом осуществляется только через `Authorization: Bearer <base64_json>`.
- **Блокировки Yandex со стороны WAF (403/400 Error)**: TurboFlare WAF автоматически блокирует входящие OAuth-запросы от Яндекса, если параметр `state` содержит закодированный callback-URL (ошибка 403 / Open Redirect protection). Также WAF может блокировать вебхуки Яндекса (Smart Home discovery, Dialogs) по признакам User-Agent.
- **Решение (Обход WAF)**: Создается отдельный субдомен (например, `oauth.yourdomain.ru`), который в DNS указывает **напрямую на IP-адрес сервера** (в обход прокси TurboFlare). Все интеграции с Яндексом (OAuth Authorize/Token URLs, Smart Home Endpoint URL, Dialogs Webhook URL) настраиваются через этот бесшовный субдомен. Nginx принимает трафик для обоих доменов и проксирует его локально.
- **Ограничение IP**: Reverse proxy маскирует только входящий трафик. Если Дом.ru забанит IP сервера, потребуется Forward Proxy/VPN на уровне ОС.

## 10. Журнал открытий и привязка к событиям

- `openings-manager.ts` записывает каждое открытие двери с типом (`manual`/`auto`) и описанием источника.
- `eventsRoutes.ts` при выдаче списка событий автоматически ищет совпадения в журнале открытий (`getOpeningByOurService`) и в снапшотах (`findSnapshotForEvent`) с допуском ±15 минут.
- Журнал и снапшоты автоматически очищаются после 30 дней.

## 11. Анонимизация

- Нигде в коде, комментариях или документации не должно быть ссылок на личные домены. Использовать `yourdomain.com`.

## 12. Архитектура плагинов и 2FA (Требуется тестирование)

- **Плагины**: Реализована базовая поддержка плагинов (папка `plugins/` и менеджер `server/plugin-manager.ts`). Плагины могут расширять функционал, добавлять собственные API-эндпоинты и управлять локальным хранилищем.
- **2FA и Распознавание лиц**: В UI вкладки «Люди» (файл `PeopleView.tsx`) реализована логика двухфакторной аутентификации (раздельные переключатели `useSchedule` и `pluginSettings.FACE_RECOGNITION`). 
- **ВАЖНО (Не протестировано)**: Архитектура была интегрирована, модальные окна обновлены, а загрузка фото переведена на новый динамический API хранилища для модулей (`/api/modules/storage/:moduleId/:key`), **однако ввиду нехватки времени полноценное тестирование этих функций в связке с SIP-открытием и реальными камерами еще не проводилось**. При последующей разработке нужно начать с верификации работоспособности этого модуля.
