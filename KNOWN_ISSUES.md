# Известные проблемы и архитектурные особенности (Known Issues)

Этот файл содержит базу знаний о проблемах, с которыми мы уже сталкивались. Обязательно к прочтению агентами перед внесением изменений в соответствующие модули.

## 1. SIP Интеграция ("Авто-открытие")
- **Ограничение на количество регистраций**: Сервер Дом.ru позволяет только ОДНУ активную SIP-регистрацию на устройство. Если наш бэкенд зарегистрируется в качестве SIP-клиента на постоянной основе, мобильное приложение пользователя перестанет получать звонки в домофон.
- **Решение**: Регистрация (REGISTER) происходит **только** при включении тумблера "Авто-открытие" и немедленно снимается (REGISTER с expires=0) после успешного открытия двери или отключения тумблера.
- **Синхронизация с базой «Люди»**: Ранее отключение или удаление карточки гостя/курьера на вкладке «Люди» не прекращало фоновые SIP-сессии, вызывая «залипание» иконок автооткрытия на вкладке камер. **Решение**: бэкенд отслеживает удаление или деактивацию карточек с префиксом `temp-` и немедленно вызывает `disableAutoOpenByDevice(deviceId)`, что завершает фоновые SIP-задачи и обновляет индикаторы на камерах в реальном времени. Индикатор активного ожидания курьеров загорается исключительно при наличии активных расписаний гостей и курьеров (расписания жильцов игнорируются, так как их прихода система не ждет).

## 2. Авторизация в API (MD5 и SHA1 хэширование)
- При авторизации (логин/пароль) используются старые протоколы провайдера: отправка двух хэшей.
- Не пытайтесь переписать логику генерации `hash1` (SHA1) и `hash2` (MD5 с кастомной солью), так как сервер Дом.ru жестко валидирует их формат. Соль жестко захардкожена в `domru-js/http/crypto.ts`.

## 3. Видеостриминг и FFmpeg
- Дом.ru отдаёт камеры в формате HLS (M3U8). Сегменты `.ts` в этих стримах часто проблемные (нет аудио или проприетарные кодеки).
- Плеер умного дома Яндекса и некоторые браузеры "падают", если в HLS потоке аудио-трек пропадает или меняется.
- **Transcoding (Video Proxy)**: Мы проксируем сегменты через локальный сервер и обрабатываем их через `ffmpeg` "на лету" (см. `server/routes/domruRoutes.ts`).
- **Осторожно с `-copyts`**: Если видео не имеет аудио, мы микшируем его с `anullsrc`. В этом случае флаг `-copyts` использовать нельзя, иначе образуется огромный разрыв (PTS gap), и HLS сломается.

## 4. Генерация временных (Гостевых) кодов
- В данный момент API Дом.ru (`getTemporalCodes`) может иметь ограничения или возвращать мок-данные (заглушки) для некоторых устройств, если услуга не подключена на аккаунте.
- Логика фронтенда не должна полностью полагаться на возможность генерации нового кода "на лету" без обработки ошибок API.
- **Работа планировщика в MSK часовом поясе**: Ранее проверки активности расписаний на сервере сбоили из-за несовпадения системного времени хоста/контейнера с реальным временем пользователя. **Решение**: бэкенд нормализует время вызова через функцию `getMskTime()`, сдвигая метку времени на UTC+3.
- **Очистка просроченных карточек**: Просроченные гостевые пропуска (ID `temp-*`), создаваемые для разовых посещений на ограниченное время, теперь автоматически вычищаются из JSON-базы пользователей на бэкенде при каждом обращении к ней (`getPeople`), чтобы не засорять хранилище и UI.

## 5. Запрет на использование личных ссылок
- Нигде в коде, комментариях или документации не должно быть ссылок на реальные сайты автора. Всегда используйте абстрактные домены типа `yourdomain.com` для обезличивания проекта.

## 6. Ограничения iFrame и UI-диалоги
- **Проблема с нативным API**: Приложение работает в iFrame-песочнице. Вызовы стандартных браузерных диалогов вроде `window.confirm()` или `window.alert()` блокируются политиками безопасности браузера или вызывают зависание фрейма.
- **Решение**: Запрещено использовать нативные всплывающие окна. Любые подтверждения опасных операций (например, удаление человека и его расписания из базы) должны реализовываться исключительно в виде анимированных кастомных React/Tailwind диалогов (`AnimatePresence` + `motion.div`), встроенных в верстку компонента.

## 7. Обработка меток времени (Timestamps) и парсинг событий
- **Формат API Дом.ru**: API (метод `/api/domru/events`) возвращает время события (`timestamp` / `occurredAt`) в виде **строки, содержащей Unix Epoch Timestamp в секундах** (например, `"1782749097"`).
- **Синхронизация времени**: При связывании событий со снимками (Snapshots) или историей открытий, бэкенд и фронтенд обязаны распознавать такие строки, переводить их в числа и умножать на 1000 (перевод в миллисекунды). 
- **Часовые пояса**: И снимки SIP (записываемые через `Date.now()`), и приходящие `timestamp` события являются абсолютными отметками в формате UTC Epoch. При поиске совпадений (например, в `findSnapshotForEvent`) достаточно использовать простую математическую разницу `Math.abs(entry.timestamp - eventTimeMs) <= maxDiffMs`. Применение деления по модулю для обхода часовых поясов категорически запрещено, так как это ломает привязку по дням и часам.

## 8. Обновления состояния интерфейса в реальном времени (WebSockets)
- **Используемое решение**: В проекте настроены и активно используются **WebSockets** (через библиотеку `socket.io` для бэкенда и фронтенда).
- **Почему не Webhooks/Polling**: Из-за ограничений среды (иногда сервер находится за NAT, иногда браузеры блокируют частые запросы), polling (`setInterval`) и входящие Webhooks для обновления UI признаны устаревшими/неэффективными.
- **Ограничение для агентов**: Если нужно передать обновление на фронтенд (например, изменился статус авто-открытия, обновился конфиг или истёк временный пропуск), используйте исключительно WebSockets (например, `broadcastAutoOpenStatusChanged` из `server/ws-manager.ts`). Не пытайтесь внедрять `setInterval` на клиенте!

## 9. Развертывание за прокси (Cloud.ru, TurboFlare)
- При работе на публичных ВМ в РФ (Cloud.ru и т.д.) провайдер может агрессивно блокировать мобильные/внешние IP-адреса пользователей по внутренним правилам Anti-DDoS.
- **Решение**: Рекомендуется прятать сервер за Reverse Proxy (TurboFlare, Cloudflare). Express в проекте уже настроен с `app.set("trust proxy", true)`.
- **Ограничение исходящего трафика**: Reverse proxy маскирует только входящий трафик к серверу. Если Дом.ru забанит сам IP-адрес сервера (ВМ), потребуется маршрутизировать исходящий трафик сервера через внешний VPN или Forward Proxy на уровне ОС (Node.js/Express сами по себе это не умеют без дополнительных пакетов вроде `https-proxy-agent`).
